## 객체지향의 사실과 오해

![](https://velog.velcdn.com/images/jjungyu12/post/9f766e7a-0067-44b0-b06d-d891b0c2ab53/image.jpg)

[객체지향의 사실과 오해](https://wikibook.co.kr/object-orientation/)를 읽고 작성한 후기입니다.

- 완독일: 2026년 1월 19일
- 저자: 조영호

### ① 책을 읽게 된 계기

Java 공부를 하면서 객체지향이 무엇인지 감으로 익히려고 했다. 하지만, 객체지향이 무엇인지 설명하라고 한다면, 못 하겠더라. 그래서 객체지향에 대해서 정확히 이해하고 다른 사람에게 설명해줄 수 있으면 좋겠다고 느껴 이 책을 선택하였다.

### ② 책 한 줄 요약

객체 지향의 핵심은 ‘클래스’가 아니라, 객체들이 주고받는 ‘메시지’와 ‘협력’에 있다.

### ③ 책의 구성과 특징

이상한 나라의 앨리스라는 동화를 메타포로 활용하여, 추상적일 수 있는 객체 지향의 개념을 쉽고 직관적으로 설명해 준다.

코드로 가득 찬 기술 책이 아니라, 개발자의 사고 방식을 확립해 주는 책이다. 

### ④ 인상 깊었던 내용

개념이 확장하고 진화하는 부분이 재밌었다.

객체지향을 이해하기 위해 사고가 어떻게 확장되어야 하는지를 단계별로 알려준다. 내가 책을 읽으면서 경험한 사고의 흐름을 정리해 보았다.


#### 1단계: **[협력하는 객체]** - 홀로 존재하는 객체는 없다
책에서는 객체를 생명체처럼 묘사한다.
객체는 공동의 목표를 달성하기 위해 서로에게 도움을 요청하고 응답하는 **‘협력적 공동체’**였다.

#### 2단계: **[행동]** - 무엇을 가졌는가가 아니라 무엇을 하는가
협력에 참여하기 위해 객체는 무엇을 해야 할까? 여기서 행동이 중요하다.
객체가 가진 데이터(상태)에 매몰되지 않고, 협력 안에서 수행하는 행동을 우선시할 때 객체의 자율성이 살아난다는 것을 알게 되었다.

#### 3단계: **[추상화]** - 복잡한 현실을 단순화하는 마법
행동을 정의하다 보면 복잡함에 부딪힌다.
책에서는 추상화를 통해 불필요한 세부 사항은 거르고 핵심 개념만 남기는 방법을 가르쳐 주었다.

#### 4단계: **[역할, 책임, 협력]** - 설계의 뼈대를 세우다
추상화된 객체들에 책임을 주고, 유사한 책임을 가진 객체들을 역할이라는 이름으로 묶는다.
즉, 책임을 갖고 있다면, 역할을 가진 어느 것이 와도 상관 없었다.
이 **‘역할, 책임, 협력’**이라는 삼박자가 갖춰질 때, 설계라는 구조가 그려졌다.

#### 5단계: **[메시지]** - 객체를 움직이는 유일한 끈
설계의 뼈대에 살을 붙이는 건 메시지였다. 객체들은 서로 내부 사정(구현)은 모른 채 오직 메시지를 통해서만 소통한다.
예를 들어, 여왕이 “저 녀석의 목을 쳐라!”라고 말했다. 이때, 여왕은 처형 집행인이 도끼를 쓰든 칼을 쓰든 상관하지 않고 **의도**만 전달할 뿐이다.
이러한 느슨한 결합은 객체지향 설계가 유연함을 유지하는 비결이라는 것을 이해할 수 있었다.

#### 6단계: **[지도]** - 변경의 파도를 견디는 구조
안정적인 구조를 중심으로 설계하는 방법을 배웠다.
기능은 빈번하게 변한다. 그래서 길을 찾는 나침반이 아니라, 지형 자체를 그려낸 지도처럼 설계해야 소프트웨어가 지속 가능하다는 비유가 정말 와 닿았다.

### ⑤ 이 책이 도움이 될 사람 / 안 맞을 사람

[추천대상]
- “객체지향 = 클래스 + 상속” 이라고 단순하게 알고 있는 개발자
- 기술 면접에서 OOP 관련 질문에 자신있게 답변을 하고 싶은 사람

[비추천 대상]
- 당장 실무에 쓸 수 있는 디자인 패턴 코드 예제가 필요한 사람
- 프로그래밍 입문자(프로그래밍 경험이 없으면 비유를 이해하기 어려울 것 같음)

### ⑥ 읽고 난 후 나의 변화

코드를 짤 때, 강의에서 하던 것처럼 ‘Entity → DTO, DAO → Service → Controller' 순으로 기계적인 코드 작성에 급급했다.

이제는 코드를 작성하기 전에 “이 객체가 어떤 책임을 져야하지?” 그리고 “누구와 협력해야 하지?”를 먼저 생각하게 되었다.
